# readme-gen

Generate a well-structured README.md for a project by analyzing its file structure, dependencies, and any existing documentation.

## When to trigger

- User asks to `create README`, `generate README`, `write readme`, or `fix the readme`
- A project directory has no `README.md`
- User says the README is outdated or missing sections

## How it works

1. **Read project context files** — `package.json` (or equivalent: `Cargo.toml`, `go.mod`, `requirements.txt`, `pyproject.toml`, `Gemfile`, etc.), `tsconfig.json`, `docker-compose.yml`, `.env.example`, and `docs/AGENTS.md` if it exists.

2. **Scan the source tree** — list the top-level and `src/` directory structure to understand architecture. Ignore `node_modules/`, `.git/`, `dist/`, `build/`, `.next/`, `.venv/`, and other typical artifact directories.

3. **Detect project characteristics**:
   - Language & runtime (TypeScript, Python, Rust, Go, Ruby, etc.)
   - Framework (Express, FastAPI, Next.js, Django, Rails, etc.)
   - Package manager (npm, pnpm, yarn, pip, cargo, bundler)
   - Presence of Docker (`Dockerfile`, `docker-compose.yml`)
   - CI/CD configs (`.github/workflows/`, `.gitlab-ci.yml`, etc.)
   - Testing setup (jest, vitest, pytest, etc.)
   - Build tooling (tsc, esbuild, webpack, vite, etc.)

4. **Generate README.md** with these sections (omit sections that don't apply):

   ```markdown
   # <project-name>

   <One-liner from package.json description or inferred purpose>

   ## Features

   - Bullet list of notable capabilities (derived from tool names, routes, package deps)

   ## Architecture

   <Auto-generated tree showing src/ structure>

   ## Getting Started

   ### Prerequisites

   <Auto-detected: Node.js version, Python version, Docker, etc.>

   ### Installation

   <Auto-generated install command based on package manager>

   ```bash
   npm install
   cp .env.example .env
   ```

   ### Running

   <Auto-generated run commands from package.json scripts, with explanations>

   ### Docker

   <Only if Dockerfile or docker-compose.yml exists>

   ```bash
   docker compose up --build
   ```

   ## Configuration

   <Auto-generated table from .env.example or env var patterns: env var | default | description>

   ## Available Tools / Scripts

   <Auto-generated table: Name | Script/Path | Description — derived from package.json scripts, MCP tool definitions, CLI entry points>

   ## Project Structure

   <Full src/ tree>

   ## Development

   <If AGENTS.md or AGENTS.md-style docs exist, include: "Adding a new X" section, common pitfalls, conventions>

   ## Testing

   <If a test framework is detected, explain how to run tests. Otherwise note that no test framework is configured.>

   ## CI/CD

   <If CI workflows are detected, describe what they do>

   ## License

   <From package.json license field>
   ```

5. **Write or overwrite** `README.md` at the project root.

## Important rules

- **Do not add a changelog or TODO section** — those should be separate files if needed.
- **Keep descriptions accurate** — only claim features/tools that actually exist in the codebase.
- **Use the project's actual package manager** in code examples (check `package.json` for lock files to determine: `package-lock.json` → npm, `pnpm-lock.yaml` → pnpm, `yarn.lock` → yarn).
- **If `.env.example` exists**, extract env var names and group them logically (like `AGENTS.md` does for this project).
- **If `docs/AGENTS.md` exists**, use it as the primary source for architecture, conventions, and "how to add X" sections. Paraphrase rather than copy verbatim.
- **Always preserve the README if it already exists** unless the user explicitly asks to regenerate it. Instead, offer a diff or suggested updates.
- **Never invent license text** — only include the License section if `package.json` has a `license` field.
